# Internationalization (i18n)

## Language Strategy

| Language | Priority | URL Pattern | Status |
|---|---|---|---|
| English | Primary | Root (`/`, `/about/`, `/portfolio/`, `/services/web-design/`, `/blog/{slug}/`) | New default |
| Croatian | Secondary | Prefixed (`/hr/`, `/hr/o-nama/`, `/hr/portfolio/`, `/hr/usluge/web-dizajn/`, `/hr/blog/{slug}/`) | Was previously root |
| German | Tertiary | Prefixed (`/de/`, `/de/uber-uns/`, `/de/portfolio/`, `/de/dienstleistungen/web-design/`, `/de/blog/{slug}/`) | Minimal content |

## The Language Flip

The old WordPress site had Croatian at root and English at `/en/`. The new site makes English the root language. This is a significant SEO change — resolved in `decisions.md` (Language URL Structure).

### Impact

- Every indexed Croatian URL (root paths) will change
- Google has hundreds of Croatian pages indexed at root
- All old `/en/` URLs redirect to root equivalents
- All old root Croatian URLs redirect to `/hr/` equivalents
- Blog posts also move from root to `/blog/` prefix (double redirect: language flip + structure change)

### Migration Plan

1. Build the new URL structure with English at root
2. Implement comprehensive redirect map (~730 rules) in `next.config.ts` `redirects()` and Next.js middleware — see `tasks/redirect-map.md`
3. Submit updated sitemap to Google Search Console
4. Add `hreflang` tags on every page to tell Google about language variants
5. Monitor Search Console for crawl errors for 3+ months

## `i18n-config.ts` Type Definition

```typescript
type Language = 'en' | 'hr' | 'de';

type RouteDefinition = {
  id: string;            // e.g. 'about', 'services-web-design', 'blog-listing'
  paths: Record<Language, string>;  // e.g. { en: '/about/', hr: '/hr/o-nama/', de: '/de/uber-uns/' }
};

type I18nConfig = {
  defaultLanguage: Language;
  languages: Language[];
  routes: RouteDefinition[];
  detectLanguage: (path: string) => Language;
  getRoute: (id: string, lang: Language) => string;
  getAlternates: (id: string) => Record<Language, string>;
};
```

## Routing Architecture

**Decision:** Explicit folders. No `[lang]` parameter. No third-party i18n library. See `decisions.md` for the formal resolution.

### Why Explicit Folders

Language-specific slugs make dynamic `[lang]` parameters unworkable. The parent segments themselves change across languages:

| EN | HR | DE |
|---|---|---|
| `/services/web-design/` | `/hr/usluge/web-dizajn/` | `/de/dienstleistungen/web-design/` |
| `/about/` | `/hr/o-nama/` | `/de/uber-uns/` |
| `/career/` | `/hr/karijera/` | `/de/karriere/` |

No `[lang]/[...slug]` pattern can unify `services`, `usluge`, and `dienstleistungen`. Third-party libraries (next-intl, next-i18next) assume a `[locale]` prefix pattern that doesn't match this URL architecture.

### How It Works

1. **Route files are thin wrappers** (5-15 lines). They declare a language constant, load the correct content file, and pass it to a shared template component.
2. **`i18n-config.ts`** is the central route map. It maps route IDs (e.g., `'services-web-design'`) to paths per language. This powers language switching, hreflang generation, and breadcrumbs.
3. **Language detection** is handled by Next.js middleware (`middleware.ts`). The middleware checks the URL path: no prefix = English, `/hr/` = Croatian, `/de/` = German. For first-time visitors (no language cookie), the middleware reads the `Accept-Language` header and redirects to the appropriate language variant. A `v2_lang` cookie persists the preference, preventing redirect loops on subsequent visits. Returning visitors with a cookie go directly to their preferred language.
4. **Translation resolution** reads the frontmatter `translations` field + the route map to produce the correct URL per language.
5. **hreflang tags** are generated automatically from the route map — no manual maintenance.
6. **Blog posts** use `[slug]` since the parent path is consistent across languages (`/blog/`, `/hr/blog/`, `/de/blog/`). `generateStaticParams` is optional — pages without it are rendered on-demand via SSR. Using it is recommended for known content (103 curated posts) to pre-render at build time for optimal first-load performance.

### Route File Pattern

Every route file follows this pattern:

```tsx
// src/app/hr/o-nama/page.tsx
import { loadPage } from '@/lib/load-pages'
import { AboutPageTemplate } from '@/components/about/about-page-template'

const LANG = 'hr' as const
const PAGE_ID = 'about'

export default function AboutPageHR() {
  const data = loadPage(PAGE_ID, LANG)
  return <AboutPageTemplate data={data} lang={LANG} />
}
```

The template component is shared across all 3 languages. Only the route wrapper changes.

## Translation Strategy

Build full trilingual infrastructure from day one. Populate English first.

1. All 387 route files exist from the start.
2. Content fallback chain: requested language -> English -> "not available" notice.
3. UI strings complete in all 3 languages from the start (~100-150 strings).
4. Blog posts: EN done (4 passes). HR/DE exist from WordPress extraction, will need rewriting later.
5. Build outputs all 387 pages regardless. Missing translations fall back at build time, not runtime.

### Pages

Most pages have all 3 language files (hr.md, en.md, de.md). German translations are often shorter or incomplete. Some pages may only have Croatian content. The fallback chain handles gaps transparently.

### Blog Posts

103 curated posts (of 190 original). English rewritten across 4 passes. Croatian and German exist from WordPress but vary in quality. They render now; rewriting is a later task.

### UI Strings

Navigation labels, button text, form labels, error messages, footer content. Stored in `src/lib/ui-strings.ts` (~120 keys × 3 languages). Complete in all languages from day one — these are short strings that can be translated upfront.

## Language Switcher

Every page needs a language switcher in the **menu panel and footer** (NOT in the header bar — see `brand-discovery.md` Section 10). It links to the equivalent page in the other language using the `translations` field from frontmatter.

If a translation doesn't exist, the switcher either hides that language option or links to the homepage in that language.

## Content Files

Each content file uses frontmatter to declare its language and translations:

```yaml
language: "hr"
translations:
  en: "about-us"     # Slug of the English equivalent
  de: "uber-uns"     # Slug of the German equivalent
```

The build process uses this to generate hreflang links and power the language switcher.

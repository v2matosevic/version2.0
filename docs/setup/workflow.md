## Build-Time Generators

Scripts in `scripts/` run during or after `npm run build` to produce static assets that the Next.js export does not generate natively.

### `scripts/generate-sitemap.ts` → `public/sitemap.xml`

Standard XML sitemap. One `<url>` per route (~387). Each entry includes `<loc>`, `<lastmod>`, `<changefreq>`, and `<priority>`. Language variants grouped via `<xhtml:link rel="alternate" hreflang="...">`. Reads the route map from `i18n-config.ts` and blog post dates from frontmatter.

### `scripts/generate-robots.ts` → `public/robots.txt`

Static output:

```
User-agent: *
Allow: /
Disallow: /dev/
Sitemap: https://version2.hr/sitemap.xml
```

### ~~`scripts/generate-htaccess.ts`~~ — Removed

No longer needed. Redirects are now handled by `next.config.ts` `redirects()` (for the ~730 301 rules) and Nginx configuration (for HTTPS enforcement, security headers, compression, and subdomain protection). The `.htaccess` generation script has been replaced by these two mechanisms.

### `scripts/generate-search-index.ts` → `public/search-index.json`

Array of objects: `{ slug, title, excerpt, category, tags, content_preview }`. `content_preview` = first 200 words of body (HTML stripped). ~50-80KB for 103 posts. Only EN posts in v1 (multilingual search is Layer 4+ work).

### `scripts/optimize-images.ts` → processes `content/**/assets/` in place

Uses Sharp to produce WebP variants at 3 srcset widths: 600w, 1200w, 1920w. Quality 80. Originals preserved. Output alongside originals with suffixes: `-600w.webp`, `-1200w.webp`, `-1920w.webp`.

## Scripts

```json
// package.json
{
  "scripts": {
    "dev": "next dev --turbopack",
    "prebuild": "tsx scripts/optimize-images.ts",
    "build": "next build",
    "postbuild": "tsx scripts/generate-sitemap.ts && tsx scripts/generate-robots.ts && tsx scripts/generate-search-index.ts",
    "start": "next start",
    "lint": "eslint .",
    "type-check": "tsc --noEmit"
  }
}
```

### Build Pipeline Order

npm lifecycle hooks handle the pipeline automatically. Running `npm run build` triggers all three stages:

1. **`prebuild`** (automatic) — `optimize-images.ts` runs first (produces WebP variants that `next build` needs)
2. **`build`** — `next build` generates the standalone server output into `.next/`
3. **`postbuild`** (automatic) — generators run sequentially after `next build` completes:
   - `generate-sitemap.ts` → writes `public/sitemap.xml`
   - `generate-robots.ts` → writes `public/robots.txt`
   - `generate-search-index.ts` → writes `public/search-index.json`

Generators run sequentially (not parallel) because they're fast (<2s each) and sequential execution simplifies error handling. If any step fails, the chain stops and the error is visible in the terminal. A failed `prebuild` aborts before `next build` runs; a failed `next build` aborts before `postbuild` runs.

## Git Setup

```bash
git init
git remote add origin https://github.com/v2matosevic/version2-site.git
```

### .gitignore

```
node_modules/
.next/
out/
.env
.env.local
.env.production
*.tsbuildinfo
.DS_Store
*.log
.vercel
.turbo
```

The `.next/` directory (build output) should not be committed. It is generated by `npm run build` and deployed to Hostinger VPS via the standalone output (`.next/standalone/`).

## Dev Workflow

1. Run `npm run dev` for local development
2. Content changes: edit files in `content/`, page hot-reloads
3. New routes: add page.tsx files in `src/app/`
4. Before committing: run `npm run type-check && npm run lint`
5. Before deploying: run `npm run build` and verify `.next/` directory
